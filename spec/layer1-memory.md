# Layer 1: Memory вЂ” Protocol Specification

**Status**: Draft v0.1 вЂ” June 2026

**Civilizational analog**: Archives, libraries, historical records

**Core question**: What does the organization remember?

---

## 1. Overview

The Memory Layer defines how organizational knowledge is represented, persisted, queried, transferred, and retired. It is the foundational layer of Agent Civilization Architecture вЂ” all other layers depend on it.

The Memory Layer is deliberately *neutral*. It records what was written, by whom, and when. It does not evaluate whether a memory is trustworthy (Layer 2), authorized (Layer 3/4), or correctly decided (Layer 5). Truth is not this layer's concern; preservation is.

---

## 2. Data Model

### 2.1 MemoryCell

A MemoryCell is the atomic unit of organizational knowledge. Every memory in the system is represented as a MemoryCell.

```json
{
  "aca_version": "0.1",
  "memory_id": "string (UUID v4, RFC 4122)",
  "status": "active | superseded | revoked | expired",

  "agent_id": "string (identifier of the owning agent)",
  "namespace": "string (organizational boundary, e.g. 'project:acme')",
  "memory_type": "fact | preference | constraint | lesson | risk",

  "content": {
    "format": "string (MIME type, e.g. 'text/plain', 'application/json')",
    "value": "string (the actual content)"
  },

  "source": {
    "type": "human | agent | system | document",
    "ref": "string (origin reference, e.g. 'session:abc', 'transfer:mem_xyz')",
    "tier": "raw_source | llm_derived | human_confirmed"
  },

  "content_hash": "string (BLAKE3 hex digest, see В§4.1)",
  "created_at": "string (ISO 8601, UTC required)",
  "created_by": "string (principal_id who performed the write)",

  "valid_until": "string | null (ISO 8601, exclusive upper bound)",
  "supersedes": "string | null (memory_id of the record this supersedes)",

  "trust_proof": "TrustProof | null (see Layer 2 В§2.2)",
  "provenance_chain": "ProvenanceChain | null (see Layer 2 В§2.3)"
}
```

**Required fields**: `aca_version`, `memory_id`, `status`, `agent_id`, `namespace`, `memory_type`, `content`, `source`, `content_hash`, `created_at`, `created_by`.

**Computed fields**: `content_hash` вЂ” MUST be computed by the implementation on write. Callers MAY provide it; implementations MUST verify and reject on mismatch.

**Optional fields**: `valid_until`, `supersedes`, `trust_proof`, `provenance_chain`.

**Removed fields**: `version` (was always 1; supersession chains are tracked via `supersedes` pointer, not version counters).

**Note on `memory_type`**: The type `decision` is intentionally absent. Decisions are Layer 5 constructs вЂ” state machines that span multiple MemoryCells вЂ” not a memory type. A decision's evidence, rationale, and outcome are stored as `fact`, `lesson`, or `constraint` memories; the decision lifecycle is managed by Layer 5.

**Note on `content_hash` scope**: The hash is computed over `content.format + ":" + content.value` as UTF-8 bytes (not `content.value` alone), ensuring that two records with the same text but different MIME types do not collide.

**Normative language**: The key words "MUST", "MUST NOT", "SHOULD", "SHOULD NOT", and "MAY" in this document are to be interpreted as described in RFC 2119 and RFC 8174.

### 2.2 AuditEvent

Every state-changing operation produces an AuditEvent. The audit log is append-only; events cannot be modified or deleted.

```json
{
  "event_id": "string (UUID v4)",
  "memory_id": "string (the affected MemoryCell)",
  "operation": "write | supersede | revoke | transfer | expire | tier_upgrade | rejected",
  "principal_id": "string (who performed the operation)",
  "timestamp": "string (ISO 8601, UTC)",
  "correlation_id": "string | null (links related events, e.g. transfer source в†” target)",
  "details": "string | null (human-readable context)"
}
```

```

```

- `active`: The memory is current and queryable.
- `superseded`: Replaced by a newer memory (the new memory's `supersedes` field points to this one).
- `revoked`: Explicitly invalidated. Soft delete вЂ” the record remains in storage for audit purposes.
- `expired`: Past its `valid_until` date. Implementations MUST treat expired records as inactive on read.

**Invariant**: Status transitions are one-way. A `superseded`, `revoked`, or `expired` record MUST NOT return to `active`.

## 3. Operations

### 3.1 Write

Creates a new MemoryCell.

**Contract**:

| Aspect | Specification |
|---|---|
| **Input** | A MemoryCell with `status: active`. `memory_id` MAY be provided or generated by the implementation. |
| **Pre-conditions** | `content.value` is non-empty. `namespace` and `agent_id` are non-empty strings. `source.tier` is one of the three defined values. |
| **Processing** | 1. Compute `content_hash` (BLAKE3 of `content.format + ":" + content.value`, UTF-8). 2. If caller provided `content_hash`, verify match; reject on mismatch. 3. Run Write Gate (see В§4). 4. Persist the record. 5. Append AuditEvent with `operation: write`. |
| **Output** | `{ memory_id, content_hash, governance_applied[] }` |
| **Failure modes** | `DuplicateMemoryError` вЂ” active record with same `content_hash` exists in same namespace. `AntiOuroborosError` вЂ” see Layer 2 spec. `NamespaceViolationError` вЂ” caller namespace mismatch. `ValidationError` вЂ” missing or invalid required fields. `HashMismatchError` вЂ” caller-provided `content_hash` does not match computed value. |

**Rejected writes**: When a write is rejected by any governance gate, implementations MUST append an AuditEvent with `operation: rejected` and include the rejection reason in `details`. This supports security monitoring and is consistent with the Mandatory Audit principle.

### 3.2 Read

Queries MemoryCells.

**Contract**:

| Aspect | Specification |
|---|---|
| **Input** | Query parameters: `memory_id` (exact lookup), or filter by `namespace`, `memory_type`, `agent_id`, `limit`, `cursor` (opaque pagination token). Text search is an OPTIONAL capability (see В§3.2.1). |
| **Pre-conditions** | None beyond valid query structure. |
| **Processing** | 1. Apply lifecycle filter: exclude `superseded`, `revoked`, and `expired` records by default. 2. Apply namespace isolation (see В§4.3). 3. Return matching records ordered by `created_at` descending, `memory_id` ascending (tie-breaker). |
| **Output** | `{ records: MemoryCell[], next_cursor: string | null }` |
| **Failure modes** | `NamespaceViolationError` вЂ” caller requests a namespace they are not authorized to read (when namespace isolation is enforced). |
| **Options** | `filterInactive: false` вЂ” include inactive records (for audit and administrative use). |

#### 3.2.1 Text Search (OPTIONAL capability)

Text search on `content.value` is an OPTIONAL capability. Implementations that support it MUST document their matching semantics (case sensitivity, Unicode normalization, tokenization). Conformance tests do not require text search.

### 3.3 Supersede

Replaces an existing memory with a new version.

**Contract**:

| Aspect | Specification |
|---|---|
| **Input** | A new MemoryCell with `supersedes` pointing to the target `memory_id`. |
| **Pre-conditions** | Target memory exists and is `active`. New memory has `status: active`. |
| **Processing** | All steps MUST be executed as a single atomic transaction: 1. Run Write Gate on the new record (including Anti-Ouroboros check вЂ” see В§4.2). 2. Set target's status to `superseded`. 3. Persist the new record. 4. Append AuditEvent `operation: supersede` for the target (with `correlation_id`). 5. Append AuditEvent `operation: write` for the new record (same `correlation_id`). If any step fails, all changes MUST be rolled back. |
| **Output** | `{ memory_id (new), superseded_id (old), governance_applied[] }` |
| **Failure modes** | All Write failure modes, plus: `AntiOuroborosError` вЂ” candidate tier is `llm_derived` AND target tier is `llm_derived` (see Layer 2 В§3.1). `TargetNotActiveError` вЂ” target is not in `active` status. |

**Concurrent supersede**: If two concurrent supersede operations target the same `active` record, the second MUST receive `TargetNotActiveError`. Implementations MUST use appropriate locking or compare-and-swap to prevent split-brain supersession.

### 3.4 Revoke

Soft-deletes a memory.

**Contract**:

| Aspect | Specification |
|---|---|
| **Input** | `memory_id` to revoke. |
| **Pre-conditions** | Target memory exists and has status `active` or `superseded`. Revoking a `revoked` record is idempotent. Revoking an `expired` record is allowed. |
| **Processing** | 1. Set target's status to `revoked`. 2. Append AuditEvent with `operation: revoke`. |
| **Output** | `{ memory_id, status: "revoked", already_revoked: boolean }` |
| **Failure modes** | `NotFoundError` вЂ” memory does not exist. |

### 3.5 Tier Upgrade

Upgrades the `source.tier` of an existing MemoryCell. This is the ONLY mechanism for tier transitions.

**Contract**:

| Aspect | Specification |
|---|---|
| **Input** | `memory_id`, `new_tier`, `trust_proof: TrustProof` (required). |
| **Pre-conditions** | Target memory exists and is `active`. `new_tier` is strictly higher than current tier (`raw_source` < `llm_derived` < `human_confirmed`). |
| **Processing** | 1. Validate tier transition is upward (reject downgrades). 2. Validate TrustProof is complete and `confirmed_by` is a valid principal. 3. For upgrades to `human_confirmed`: `confirmed_by` MUST be a human principal (not an agent). 4. Update target's `source.tier` to `new_tier`. 5. Set target's `trust_proof` to the provided TrustProof. 6. Append to target's `provenance_chain` a `tier_upgrade` transition. 7. Append AuditEvent with `operation: tier_upgrade`. |
| **Output** | `{ memory_id, old_tier, new_tier }` |
| **Failure modes** | `TierDowngradeError` вЂ” `new_tier` is not strictly higher than current tier. `InvalidTrustProofError` вЂ” TrustProof is missing or incomplete. `NonHumanConfirmerError` вЂ” upgrade to `human_confirmed` but `confirmed_by` is not a human principal. `NotFoundError` / `TargetNotActiveError`. |

**Note**: Tier upgrade does NOT trigger the dedup gate (В§4.1). The content is unchanged; only the trust pedigree changes. This resolves the dedup-vs-tier-upgrade conflict identified in review.

### 3.6 Transfer

Copies a memory to another namespace and/or agent.

**Contract**:

| Aspect | Specification |
|---|---|
| **Input** | `memory_id` (source), `target_namespace`, `target_agent_id`. |
| **Pre-conditions** | Source memory exists and is `active`. Caller has read access to source namespace and write access to target namespace. |
| **Processing** | 1. Create a new MemoryCell in the target namespace with: new `memory_id`, `agent_id: target_agent_id`, `namespace: target_namespace`, `source.ref: "transfer:{original_memory_id}"`, `source.tier` preserved from original (tier does NOT upgrade on transfer). 2. Run Write Gate on the new record. 3. Append AuditEvent with `operation: transfer` for both source and new records. 4. Source record remains `active` (transfer is copy, not move). |
| **Output** | `{ new_memory_id, source_memory_id, governance_applied[] }` |
| **Failure modes** | All Write failure modes, plus: `NamespaceViolationError` вЂ” caller lacks permission to write to target namespace. `SourceNotActiveError` вЂ” source memory is not `active`. |

### 3.7 Expire

Explicitly transitions an active memory to expired status. Implementations MAY also lazily expire records on read when `valid_until` is past.

**Contract**:

| Aspect | Specification |
|---|---|
| **Input** | `memory_id` to expire, OR a batch scan with no input (system-triggered expiration of all records past `valid_until`). |
| **Pre-conditions** | Target memory exists and is `active`. For explicit expire: no condition on `valid_until`. For system-triggered: `valid_until` is non-null and in the past. |
| **Processing** | 1. Set target's `status` to `expired`. 2. Append AuditEvent with `operation: expire`. |
| **Output** | `{ memory_id, status: "expired" }` |
| **Failure modes** | `NotFoundError` / `TargetNotActiveError`. |

**Dual mechanism**: Expiration can occur in two ways: (1) explicit `expire` operation call, or (2) lazy evaluation at read time when `valid_until` is past. Both result in the same effective behavior (record excluded from default reads). However, only the explicit operation updates stored `status` and writes an audit event. Implementations SHOULD run periodic expiration sweeps to keep stored status consistent with effective status.

### 3.8 Audit

Retrieves the append-only event log for a memory.

**Contract**:

| Aspect | Specification |
|---|---|
| **Input** | `memory_id`. |
| **Pre-conditions** | None. Audit is accessible even for revoked/expired records. |
| **Processing** | Return all AuditEvents for the given `memory_id`, ordered by `timestamp` ascending. |
| **Output** | `AuditEvent[]` |
| **Failure modes** | `NotFoundError` вЂ” no events exist for this memory_id. |

## 4. Governance Gates

The Write Gate is a pipeline of checks that runs on every write, supersede, and transfer operation. Gates are organized into two conformance profiles:

- **Layer 1 conformant**: Dedup (В§4.1) + Namespace Isolation (В§4.3) + Lifecycle Filter (В§4.4). These are pure memory-layer gates.
- **Layer 1+2 conformant**: All Layer 1 gates PLUS Anti-Ouroboros (В§4.2). This gate enforces a Layer 2 trust rule at the write path, and is therefore a cross-layer gate.

Implementations MUST enable all gates by default. An implementation MAY allow disabling the Anti-Ouroboros gate (dropping from L1+2 to L1 conformance), but MUST NOT allow disabling Dedup, Namespace Isolation, or Lifecycle Filter while claiming any conformance level.

### 4.1 Content-Hash Deduplication

**Rule**: If an `active` record with the same `content_hash` AND same `memory_type` exists in the same `namespace`, reject the write.

**Algorithm**: BLAKE3 hash of `content.format + ":" + content.value` (UTF-8 encoded bytes).

**Exemption**: Tier Upgrade operations (В§3.5) are exempt from dedup вЂ” the content is unchanged, only the trust pedigree changes.

**Failure**: `DuplicateMemoryError { existing_memory_id, content_hash }`

**Test vectors** (BLAKE3, UTF-8):
- Input: `text/plain:Hello, world!` в†’ `a0b4f4a45a1a ...` (implementations MUST match reference)
- Input: `text/plain:` (empty value) в†’ (reject: content.value is non-empty precondition)

### 4.2 Anti-Ouroboros (cross-layer gate, delegated to Layer 2)

See [Layer 2: Trust](layer2-trust.md) В§3.1.

**Direction convention**: In the supersede operation, the *candidate* is the new record being written, and the *target* is the existing record being superseded. The Anti-Ouroboros rule checks: `candidate.source.tier == llm_derived AND target.source.tier == llm_derived в†’ REJECT`.

### 4.3 Namespace Isolation

**Rule**: When namespace isolation is enabled, a caller MUST have an associated `caller_namespace`. Writes are restricted to the caller's own namespace. Reads default to the caller's namespace unless an explicit cross-namespace read is granted.

**Failure**: `NamespaceViolationError { requested_namespace, caller_namespace }`

### 4.4 Lifecycle Filter

**Rule**: Read operations MUST exclude `superseded`, `revoked`, and `expired` records by default. Records with `valid_until` in the past MUST be treated as `expired` regardless of their stored `status`.

**Override**: `filterInactive: false` returns all records including inactive ones. This is intended for audit and administrative operations only.

## 5. Storage Requirements

The Memory Layer is storage-agnostic. Implementations MAY use any persistence backend (SQLite, PostgreSQL, JSON files, HTTP APIs, etc.) as long as the following properties are maintained:

1. **Durability**: A successfully written MemoryCell MUST survive process restart.
2. **Atomicity**: A write-and-audit pair MUST be atomic. If the write succeeds but the audit append fails, both MUST be rolled back.
3. **Append-only audit**: The audit log MUST NOT support update or delete operations.
4. **Content-hash index**: The store MUST support efficient lookup by `content_hash` within a namespace (required for deduplication).

---

## 6. Wire Format

MemoryCells are serialized as JSON (RFC 8259). Implementations MUST accept and produce UTF-8 encoded JSON. Field ordering is not significant. Unknown fields MUST be preserved on read and re-emitted on write (forward compatibility).

The canonical form for content-hash computation is `content.format + ":" + content.value` as raw UTF-8 bytes (see В§4.1), not the JSON-serialized form of the MemoryCell.

---

## 7. Conformance

An implementation is **ACA Layer 1 conformant** if it:

1. Correctly serializes and deserializes MemoryCell and AuditEvent structures.
2. Implements all eight operations (write, read, supersede, revoke, tier_upgrade, transfer, expire, audit) with the specified contracts.
3. Enforces Layer 1 governance gates (dedup В§4.1, namespace isolation В§4.3, lifecycle filter В§4.4) by default.
4. Maintains append-only audit log integrity, including `operation: rejected` events for denied writes.
5. Passes the Layer 1 conformance test suite (forthcoming).

An implementation is **ACA Layer 1+2 conformant** if it additionally:

6. Enforces the Anti-Ouroboros gate (В§4.2) as a mandatory write-path check.
7. Satisfies all Layer 2 conformance criteria (see [Layer 2 В§5](layer2-trust.md#5-conformance)).
